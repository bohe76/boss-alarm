# 보스 알리미 (Boss Alarm) — 제품 요구사항 문서 (PRD)

> 버전: v3.0.0 기준 | 최초 작성: 2026-04-19 | 상태: 초안

---

## 1. 비전

> "길드원 누구나 보스 알림을 놓치지 않는다."

모바일 MMORPG(오딘, 리니지 M/W 등)에서 보스 출현 시간을 관리하는 기존 방법(디스코드 봇, 카카오톡 수동 공유)은 CLI 지식이 필요하거나 공유 비용이 크다. 보스 알리미는 설치 없이 브라우저에서 즉시 사용 가능한 GUI 기반 PWA로, 길드 단위 보스 알림 관리를 누구에게나 직관적으로 제공한다.

---

## 2. 타깃 사용자 (페르소나)

### 페르소나 A — 길드 마스터 "정보 관리자"

| 항목 | 내용 |
|------|------|
| 역할 | 오딘/리니지 M 길드 마스터 또는 임원 |
| 핵심 목표 | 길드 전체의 보스 시간표를 빠르게 정리하여 공유 |
| 주요 행동 | 보스 스케줄러에서 오늘 보스 시간 입력 → 시간표 이미지 내보내기 → 카카오톡 공유 |
| 페인 포인트 | 디스코드 봇은 CLI 명령어가 복잡하고, 매번 수동 입력·공유가 번거롭다 |
| 환경 | 모바일 우선 (Android/iOS), 게임 플레이 중 병행 사용 |

### 페르소나 B — 일반 길드원 "알림 수신자"

| 항목 | 내용 |
|------|------|
| 역할 | 보스 공격에 참여하는 일반 길드원 |
| 핵심 목표 | 공유된 URL을 열어 보스 시간을 확인하고, 직접 알림을 받는다 |
| 주요 행동 | 공유 URL 접속 → 보스 시간표 확인 → 브라우저 알림 대기 |
| 페인 포인트 | 길드 채팅방의 텍스트를 놓치거나, 보스 시간을 일일이 외워야 한다 |
| 환경 | 모바일 우선, 카카오톡 인앱 브라우저 진입 빈도 높음 |

## 3. 시장 맥락

- **주요 경쟁 대안**: 디스코드 봇(CLI 기반), 카카오톡 수동 입력, 엑셀/노션 시간표
- **차별점**: 설치 불필요한 PWA, GUI 전용 입력, 이미지/URL 한 번에 공유, LocalStorage 완결형(서버 없음)
- **배포 채널**: GitHub Pages (https://bohe76.github.io/boss-alarm)
- **현재 상태**: v3.0.0 출시 (2026-04-19), 132개 테스트 통과

---

## 4. 핵심 가치 제안

| 가치 | 설명 |
|------|------|
| 즉시 사용 | 설치 없이 URL 접속만으로 사용 가능 (PWA) |
| GUI 전용 | CLI 없이 폼 입력만으로 보스 스케줄 관리 |
| 공유 최적화 | 이미지 내보내기 + URL 공유로 카카오톡 공유 1단계 완료 |
| 알림 정확도 | Web Worker 기반 백그라운드 타이머, 5분/1분/0분 3단계 알림 |
| 프라이버시 | 모든 데이터는 LocalStorage 전용 — 외부 서버에 데이터 전송 없음 |

## 8. 의존성 및 제약

| 항목 | 내용 |
|------|------|
| 배포 인프라 | GitHub Pages (정적 호스팅, 서버 없음) |
| 외부 API | TinyURL API (URL 단축, 실패 시 원본 URL 폴백) |
| 브라우저 지원 | PiP 기능은 Chromium 계열 최신 버전 필요 |
| 데이터 격리 | LocalStorage 전용 — 인터넷 없이도 동작, 기기 간 동기화 없음 |
| 테스트 | Vitest + JSDOM (132 tests, npm test) |
| 언어/플랫폼 | 바닐라 JS (ES Modules), 빌드 스텝 없음 |

## 9. 리스크

| 리스크 | 영향 | 대응 방안 |
|--------|------|-----------|
| LocalStorage 용량 초과 | 데이터 저장 실패 | `DB.save()` QuotaExceededError 감지 후 사용자 알림 |
| 브라우저 백그라운드 타이머 지연 | 알림 부정확 | Web Worker 도입으로 완화 (v2.17.6+) |
| 카카오톡 인앱 브라우저 제한 | 기능 미동작 | 인앱 브라우저 감지 후 외부 브라우저 유도 |
| 광고 UX 저하 | 이탈률 증가 | 광고 배치 설계 시 사용자 경험 우선 검토 필요 (TBD) |
| 멀티유저 전환 시 아키텍처 전면 개편 | 개발 비용 대규모 증가 | 서버 도입 시 별도 설계 필요 |
