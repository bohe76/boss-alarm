### `docs/system_module_details.md` 문서의 보완할 점

`docs/system_module_details.md`는 애플리케이션의 각 JavaScript 모듈에 대한 상세한 설명을 제공하는 중요한 문서입니다. 코드 학습을 바탕으로 볼 때, 다음 영역에서 보완이 필요합니다.

---

#### 1. API 명세의 일관성 및 완전성 강화

*   **현재 상태:** 일부 모듈의 `export` 함수는 상세하게 설명되어 있지만, 모든 `export` 함수 및 변수에 대한 인자, 반환 값, 예상되는 부작용 등이 일관된 형식으로 명세되어 있지 않습니다.
*   **보완점:** 모든 `export` 항목에 대해 JSDoc과 같은 표준화된 형식을 사용하여 다음 정보를 명확하게 기술해야 합니다.
    *   함수/변수의 목적
    *   모든 매개변수 (이름, 타입, 설명, 필수/선택 여부)
    *   반환 값 (타입, 설명)
    *   발생 가능한 에러 또는 예외
    *   모듈의 내부 상태 변경 또는 DOM 조작과 같은 부작용
*   **예시:** `CustomListManager`와 같이 `export` 객체 형태로 제공되는 경우, 각 메서드에 대한 명세를 추가해야 합니다.

#### 2. 핵심 내부 로직 설명의 심층화

*   **현재 상태:** "핵심 내부 로직" 섹션이 일부는 간결하게 잘 요약되어 있지만, 일부는 코드의 동작을 단순히 나열하는 수준에 그치거나 충분한 맥락을 제공하지 못합니다.
*   **보완점:**
    *   단순한 "무엇을 하는지"를 넘어 "왜 그렇게 하는지", "어떤 디자인 패턴을 따르는지", "어떤 제약 조건을 해결하는지"와 같은 설계 의도를 포함해야 합니다.
    *   특히 복잡한 로직(예: `alarm-scheduler.js`의 `checkAlarms` 로직)의 경우, 주요 분기점, 데이터 변환 단계, 상태 관리 흐름 등을 의사 코드나 핵심 단계별 설명으로 더 명확하게 제시할 필요가 있습니다.

#### 3. 모듈 간 상호 작용 설명 구체화

*   **현재 상태:** 모듈 간 상호 작용이 언급되어 있으나, 특정 함수 호출 또는 이벤트 발행이 어떤 목적을 가지고 어떤 데이터를 전달하는지에 대한 구체적인 설명이 부족합니다.
*   **보완점:**
    *   모듈이 다른 모듈의 함수를 직접 호출하는 경우 (예: `event-handlers.js`에서 `parseBossList` 호출), 호출의 트리거 및 예상되는 결과에 대한 설명을 강화해야 합니다.
    *   `EventBus`를 통한 상호 작용의 경우, 어떤 이벤트가 발행되고(emit), 어떤 데이터가 전달되며, 해당 이벤트를 어떤 모듈이 수신하여(on) 어떤 동작을 수행하는지에 대한 설명을 추가하여 느슨한 결합의 흐름을 명확히 해야 합니다.

#### 4. 코드 불일치 해소 및 문서 업데이트

*   **현재 상태:** `event-handlers.js`가 애플리케이션의 메인 진입점이라고 명시되어 있으나, 실제 코드에서는 `src/app.js`가 유사한 역할을 수행하고 있어 문서와 코드 간의 불일치가 존재합니다.
*   **보완점:**
    *   `src/app.js`와 `src/event-handlers.js` 중 어느 것이 메인 오케스트레이터 역할을 하는지 명확히 하고, 문서에 해당 파일 이름을 정확히 반영해야 합니다. (이전 `architecture_gaps_report.md`에서 지적된 사항과 연결됨)
    *   문서에 언급된 `initNotificationSettingsScreen` 함수는 `src/screens/notifications.js`에 존재하며, `event-handlers.js` 내에 분산 처리되고 있다는 내용이 `system_data_flow.md`에 있었는데, `system_module_details.md`에는 `src/screens/notifications.js`가 화면 모듈 그룹에 포함되어 있어 혼란을 야기합니다. 실제 `src/screens/notifications.js` 파일이 존재하는 것으로 확인되었으므로, 문서 간의 일관성을 유지하고 정확한 정보를 제공해야 합니다.

### `docs/system_module_dependencies.md` 문서의 보완할 점

`docs/system_module_dependencies.md`는 모듈 간 의존성 관계를 설명하며, 느슨한 결합을 강조하고 있습니다. 다음 영역에서 보완이 필요합니다.

---

#### 1. 시각적 의존성 다이어그램 추가

*   **현재 상태:** 텍스트 기반의 흐름도와 표 형식으로 의존성을 설명하고 있습니다.
*   **보완점:**
    *   UML 컴포넌트 다이어그램, 계층형 의존성 그래프, 또는 C4 모델과 같은 시각적 다이어그램을 추가하여 모듈 간의 관계를 한눈에 파악할 수 있도록 해야 합니다. 특히 EventBus를 통한 느슨한 결합 관계를 시각적으로 표현하면 이해도를 크게 높일 수 있습니다.
    *   이를 통해 복잡한 의존성 구조를 더 직관적으로 이해할 수 있습니다.

#### 2. 의존성 유형의 명확화

*   **현재 상태:** 모듈이 다른 모듈에 의존하는 방식(직접 호출, `EventBus.emit`, 데이터 접근 등)이 혼재되어 있습니다.
*   **보완점:**
    *   표에서 "호출하는 주요 함수 / 발행하는 이벤트" 항목을 "의존성 유형"과 "세부 내용"으로 분리하여 의존성 유형(예: 직접 함수 호출, EventBus 이벤트 발행, 데이터 관리자 접근, UI 렌더링 함수 사용 등)을 명확히 제시해야 합니다.
    *   특히 `EventBus.emit(...)`의 경우, 어떤 이벤트가 발행되는지 구체적인 이벤트 이름을 명시해야 합니다.

#### 3. 양방향/단방향 의존성 강조 및 표현

*   **현재 상태:** 문서 내에서 "상위 -> 하위 단방향 흐름" 및 "느슨한 결합"을 지향한다고 언급하지만, 표에서 이러한 원칙이 시각적으로 명확하게 드러나지 않을 수 있습니다.
*   **보완점:**
    *   시각적 다이어그램을 통해 단방향 흐름과 EventBus를 통한 간접적인 상호 작용을 명확하게 구분하여 표현해야 합니다.
    *   각 모듈이 다른 모듈로부터 받는 이벤트나 데이터에 대한 정보(즉, 구독하는 이벤트)도 함께 명시하여 전체적인 상호 작용의 맥락을 제공해야 합니다.

#### 4. 코드 변경에 따른 문서 업데이트의 일관성 유지

*   **현재 상태:** "리팩토링 v4.0 아키텍처의 주요 의존성 관계는 **`app.js`가 각 화면 모듈을 초기화**하고..."라는 문장과 "리팩토링 v4.0 아키텍처의 주요 의존성 관계는 **`event-handlers.js`의 `initApp` 함수가 핵심 모듈들을 초기화**하고..."라는 두 문장이 존재하여 혼란을 야기합니다.
*   **보완점:**
    *   `src/app.js`와 `src/event-handlers.js`의 역할 불일치를 해소하고, 메인 오케스트레이터의 파일 이름을 문서 전체에서 일관되게 사용해야 합니다. (이전 `architecture_gaps_report.md`에서 지적된 사항과 연결됨)

### `docs/system_data_flow.md` 문서의 보완할 점

`docs/system_data_flow.md`는 애플리케이션의 주요 데이터 흐름을 상세히 설명하는 중요한 문서입니다. 코드 학습을 바탕으로 볼 때, 다음 영역에서 보완이 필요합니다.

---

#### 1. 데이터 흐름 다이어그램 (DFD 또는 시퀀스 다이어그램) 추가

*   **현재 상태:** 각 시나리오별로 텍스트 기반의 단계별 설명을 제공하고 있습니다. 이는 상세하지만, 전체적인 흐름을 한눈에 파악하기 어렵습니다.
*   **보완점:**
    *   각 주요 데이터 흐름 (예: "애플리케이션 초기 로드", "알람 시작 및 주기적 갱신")에 대한 데이터 흐름 다이어그램(DFD) 또는 시퀀스 다이어그램을 추가하여 시각적 이해도를 높여야 합니다.
    *   특히 모듈 간의 데이터 전달, EventBus를 통한 이벤트 발생 및 구독, 데이터 저장소(LocalStorage)와의 상호 작용 등을 명확하게 표현해야 합니다.

#### 2. 데이터 내용의 구체성 강화

*   **현재 상태:** 데이터가 전달되거나 상태가 변경될 때, "보스 목록", "고정 알림"과 같이 추상적인 용어로 설명되는 경우가 있습니다.
*   **보완점:**
    *   각 단계에서 어떤 *구체적인 데이터 구조나 형식*이 전달되거나 변경되는지 (예: `BossDataManager.bossSchedule`은 `Boss` 객체 배열, `LocalStorageManager`는 JSON 문자열) 더욱 명확하게 기술해야 합니다.
    *   데이터 유효성 검사 및 변환이 이루어지는 지점을 명시하고, 그 과정에서 데이터가 어떻게 변화하는지 설명해야 합니다.

#### 3. 상태 변경 및 동기화 메커니즘 명확화

*   **현재 상태:** `BossDataManager`나 `LocalStorageManager`와 같은 전역 상태 관리자가 업데이트된다고 언급하지만, 정확히 어떤 속성이 어떤 값으로 변경되는지에 대한 상세 설명이 부족합니다.
*   **보완점:**
    *   특정 이벤트나 사용자 액션이 발생했을 때, 어떤 데이터 관리자의 어떤 상태가 어떻게 변경되고, 이 변경이 다른 모듈에 어떻게 파급되는지 (예: `EventBus`를 통해 UI 갱신을 트리거) 더 자세히 설명해야 합니다.
    *   특히 `alarm-scheduler.js`의 `checkAlarms()` 함수가 `BossDataManager`의 `nextBoss` 정보를 업데이트하는 과정과 `EventBus.emit('refresh-dashboard')`를 통해 대시보드가 갱신되는 연계 흐름을 더욱 명확히 할 수 있습니다.

#### 4. 오류 처리 및 엣지 케이스 시나리오 추가

*   **현재 상태:** 일반적인 성공 케이스 위주로 데이터 흐름을 설명하고 있습니다.
*   **보완점:**
    *   데이터 파싱 실패, API 호출 실패 (예: TinyURL API), 사용자 입력 유효성 검사 실패 등 발생 가능한 오류 시나리오와 그에 따른 데이터 흐름 및 사용자 피드백 (예: `log()`, `showToast()`)에 대한 설명을 추가해야 합니다.
    *   엣지 케이스 (예: 빈 보스 목록, 잘못된 URL 파라미터)에 대한 처리 방식도 포함하면 문서의 완성도를 높일 수 있습니다.

#### 5. 코드 변경에 따른 문서 업데이트의 일관성 유지

*   **현재 상태:** `3.10. 알림 설정 화면 (`src/screens/notifications.js`)` 섹션과 `3.11. (`src/screens/index.html`에 존재하지 않는) `notifications.js` 파일에 대한 주석` 섹션이 존재하여, `src/screens/notifications.js` 파일이 실제로 존재하는지, 아니면 존재하지 않는 파일에 대한 설명인지 혼란을 줍니다. 실제 `src/screens/notifications.js` 파일이 존재하는 것으로 확인되었으므로, 해당 섹션을 통합하고 정확한 정보로 수정해야 합니다. (이전 `architecture_gaps_report.md`에서 지적된 사항과 연결됨)
*   **보완점:**
    *   `src/event-handlers.js`의 `initApp` 함수 내에서 `DOM.bossListInput.value`에 기본 보스 목록을 설정하는 로직이 있는데, 이 과정에서 `formatMonthDay` 유틸리티를 사용하여 날짜를 삽입하는 상세 로직이 문서에는 간략하게 설명되어 있습니다. 이 부분의 데이터 생성 및 변환 로직을 더 구체적으로 기술할 필요가 있습니다.
*   **코드 변경에 따른 문서 업데이트:** `system_data_flow.md`는 "리팩토링 v4.0 아키텍처의 주요 의존성 관계는 **`event-handlers.js`의 `initApp` 함수가 핵심 모듈들을 초기화**하고..."로 시작하지만, `app.js`와 `event-handlers.js`의 역할 혼동이 있었습니다. 문서에서는 `event-handlers.js`를 기준으로 설명하고 있으나, 코드 실제 구현은 `app.js`에서 이루어졌던 점을 명확히 하고, 문서의 일관성을 확보해야 합니다. (이전 `architecture_gaps_report.md`에서 지적된 사항과 연결됨)
